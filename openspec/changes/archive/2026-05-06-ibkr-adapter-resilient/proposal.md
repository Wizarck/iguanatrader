## Why

Slice T1 (`trading-models-interfaces`) shipped the `BrokerPort` Protocol on 2026-05-06 — five async methods (`place_order`, `cancel_order`, `reconcile_fills`, `get_position`, `get_account_equity`) with no implementation. Wave 3 cannot deliver any live behaviour until a concrete adapter satisfies that contract: T4 (`trading-routes-and-daemon`) cannot wire `execute_on_approval` to a real broker, K1's kill-switch cannot enforce `NFR-R7` against an actual TWS, and slice 0 `shared-primitives` shipped `HeartbeatMixin` + canonical backoff `[3, 6, 12, 24, 48]` (NFR-R7) on 2026-05-01 explicitly waiting for this slice as their first consumer. **Live broker resilience is the highest-stakes invariant in iguanatrader**: TWS Gateway crashes mid-session, the home-router drops at 03:00, and Windows pushes a forced reboot at 04:00 patch-Tuesday — the bot MUST survive these without losing fills, double-submitting orders, or stalling the propose→execute loop. This slice plants `IBKRAdapter` over `ib_async` (the maintained MIT-licensed fork of `ib-insync`) with: (a) a 30s heartbeat against TWS via `client.reqCurrentTime()` (NFR-P8), (b) the canonical backoff loop wrapped to terminate at 5 attempts → emit `KillSwitchTripped` (NFR-R7), (c) reconciliation-on-reconnect that diffs `client.openOrders()` + `client.executions(since=last_seen)` against the local `orders` + `fills` tables and emits `broker.fill.catchup` events for any rows the adapter missed during the outage (FR16 + NFR-R2), (d) idempotent `place_order` keyed by client-generated `client_order_id` (NFR-I1) so a retry after partial-network-drop never produces a duplicate broker submission, and (e) `ibkr_brokerage_model.py` modelling IBKR-specific quirks (paper port 7497 vs live 7496, supported `order_type` whitelist, market-data subscription gating, commission model) so the adapter stays clean. Tests use `ib-insync-mock` (or an equivalent in-tree fake of the `ib_async` `IB` client surface) — TWS is NEVER required in CI; the integration tests exercise the resilience contract deterministically. Now is the right time because T1 is archived (frozen contract), `shared-primitives` is archived (frozen mixin + backoff), and Wave 3 has T2 + T3 running in parallel under disjoint write paths (`brokers/` vs `strategies/`).

## What Changes

- **New broker module `contexts/trading/brokers/`** — bounded-context-internal package with two concrete files: `ibkr_adapter.py` (the `IBKRAdapter` class implementing `BrokerPort` + composing `HeartbeatMixin`) and `ibkr_brokerage_model.py` (a frozen-dataclass model of IBKR-specific configuration: paper/live port choice, supported order types, market-data permission flags, commission schedule). Empty `__init__.py` re-exports `IBKRAdapter` for `service.py` consumption.
- **`IBKRAdapter` class** — implements all five `BrokerPort` methods over `ib_async`. Inherits `HeartbeatMixin` from `shared-primitives` and overrides `_send_heartbeat` (calls `client.reqCurrentTime()` with 90s deadline per NFR-P8) and `_on_disconnect` (emits structlog + publishes `broker.connection.lost` event). Exposes `connect()` / `disconnect()` lifecycle methods + a `reconcile_state()` helper called automatically after every successful reconnect.
- **Idempotency layer** — every `place_order(NewOrder)` call writes the order row to `orders` table with status `pending` keyed by `client_order_id` (UUIDv4 generated by the service before invoking the adapter) BEFORE issuing `client.placeOrder()`. If the adapter re-receives the same `NewOrder` (after reconnect or service-side retry), it looks up `client_order_id` in `orders` and returns the existing `BrokerOrderId` without a new broker submission. Aligns with NFR-I1 + FR15 lifecycle.
- **Reconciliation flow** — on reconnect (after `HeartbeatMixin.reconnect_loop` returns successfully), `IBKRAdapter._post_reconnect_reconciliation()` runs: queries `client.openOrders()` + `client.reqExecutions(ExecutionFilter(time=last_disconnect_at_utc.strftime('%Y%m%d-%H:%M:%S')))`, diffs against local `orders` (open) + `fills` (since `last_disconnect_at`), emits `broker.fill.catchup` event for each unobserved fill (consumed by `service.py::reconcile_fills_handler` to INSERT the row + update trade state). FR16 + NFR-R2 surface area. Documented gotcha: `client.openOrders()` returns only orders submitted by the current API session; `client.reqAllOpenOrders()` queries TWS-wide state including manually-placed orders — the adapter uses `reqAllOpenOrders()` to catch the case of a TWS restart that wiped session-scoped state.
- **Backoff + kill-switch escalation** — `HeartbeatMixin.reconnect_loop` is unbounded by design (slice 2 D6); `IBKRAdapter` wraps it with `_resilient_reconnect_loop()` that increments an attempt counter and, on attempt 6 (after walking `[3, 6, 12, 24, 48]`), publishes `KillSwitchTripped(reason="ibkr.adapter.reconnect_exhausted", ...)` to the MessageBus and stops further attempts. K1's kill-switch handler halts trading per NFR-R7.
- **`ibkr_brokerage_model.py`** — `IBKRBrokerageModel` frozen dataclass with: `mode: Literal["paper","live"]`, `host: str`, `port: int` (default 7497 for paper, 7496 for live), `client_id: int`, `supported_order_types: frozenset[str]` (whitelist: `MKT`, `LMT`, `STP`, `STP LMT`; reject anything else with `IntegrationError`), `market_data_subscriptions: dict[str, bool]` (per-symbol flag — TRUE means an active paid sub exists; FALSE means snapshot-only with delayed quotes), `commission_model: Literal["tiered","fixed"]` (informational; commissions come from the fill event itself). Helper `from_settings(settings: Settings) -> IBKRBrokerageModel` reads from the slice-2 settings layer.
- **Connection lifecycle structured events** — emit `broker.connection.established` (after first successful connect), `broker.connection.lost` (on disconnect detection), `broker.connection.restored` (after successful reconnect), `broker.fill.received` (live fill stream), `broker.fill.catchup` (reconciled missed fill). Names follow `<context>.<entity>.<action>` per NFR-O8. Each carries `tenant_id`, `correlation_id`, `mode` ("paper"/"live"), `iso8601_utc` timestamp.
- **Mock-based integration tests** — `tests/integration/test_ibkr_resilience.py` (clean connect → heartbeat for 90s simulated → mid-flight disconnect → reconnect succeeds at attempt 2 → state restored) and `tests/integration/test_reconciliation.py` (place 2 orders → simulate disconnect → broker fills 1 of them during outage → reconnect → assert exactly 1 `broker.fill.catchup` event published with correct payload). Both use `ib-insync-mock` (or an in-tree fake harness if the lib's `ib_async` compat is incomplete; verify in design D7).
- **No strategy logic, no routes, no CLI, no frontend, no daemon** — strategies live in T3, route bodies + CLI + frontend live in T4. This slice is the broker plumbing only.
