## Why

Slice O1 (`observability-cost-meter`) just shipped the cost-meter / budget / structlog primitives. Slices K1 (risk), T1 (trading-models), R1 (research-bitemporal), and P1 (approval-channels) are landing alongside in Wave 2 — they emit events into the MessageBus (`risk.kill_switch.tripped`, `trading.fill.received`, `research.fact.added`, `approval.proposal.timed_out`), but **nobody is listening on a schedule, and no human is consuming digests of what happened.** The MVP's defining UX (PRD Journey 3, Journey 4) is **timed routines that wake up before/during/after the market**: a premarket briefing at 06:30 ET that reads overnight news + earnings calendar + insider buys and synthesizes a watchlist brief; a midday pulse at 12:30 ET that summarizes open positions + risk dashboard; a postmarket summary at 16:30 ET that closes out P&L + refreshes the next-day watchlist; a Sunday weekly review at 18:00 ET that emits a PDF with equity curve + trade-by-trade narrative + lessons + outlook (FR44). Without slice O2, none of those routines exist — the bot sits silent between approval requests, and Arturo never gets a "good morning, here's what to watch today" message. Humans cannot reliably check the market four times a day forever; **systematic discipline is exactly what the bot must provide**, and the weekly review is the retrospective journaling layer that closes the loop.

The slice ALSO carries the **tier-1 alert filter** (FR33 + NFR-P3) — the "wake the human up immediately" path for events like `risk.kill_switch.tripped`, FDA approvals on watchlist symbols, insider 10%+ buys, IBKR connection lost. Tier-1 emits via P1's Telegram/Hermes channels within 60s p99; tier-2 events land in the next routine digest; tier-3 events live in the audit log only. Now is the right time because (a) O1 just shipped cost_meter so each routine can budget-gate its LLM calls before invoking Sonnet/Haiku, (b) K1 just shipped `kill_switch_events` that the alert filter subscribes to, (c) R1 just shipped `research_facts` that the routines query for "what changed overnight", (d) Wave 3 needs the scheduler infrastructure landed before Wave 4's `trading-routes-and-daemon` (T4) can wire its end-to-end happy path. The slice plants APScheduler with `SQLAlchemyJobStore` (NOT memory store — long-uptime memory leak risk) + four LangGraph workflow definitions + the alert filter rules engine + the report PDF generator (reportlab).

## What Changes

- **Bounded context `orchestration`** — new `apps/api/src/iguanatrader/contexts/orchestration/` package with `__init__.py`, `service.py` (lifecycle: register jobs at startup, wire MessageBus subscriptions for tier-1 alerts, expose `run_routine_now(name)` for manual trigger), `scheduler.py` (APScheduler `AsyncIOScheduler` instance with `SQLAlchemyJobStore` backed by the same engine + `timezone="America/New_York"` ET-aware crons + DST-aware via `pytz`/`zoneinfo`), `alert_filter.py` (rule engine matching incoming MessageBus events to a tier 1/2/3 classification + emitting tier-1 to P1 channels), `tier1_alerts.py` (canonical tier-1 rule table — `kill_switch.tripped`, `ibkr.disconnected`, `fda.approval`, `insider.buy_10pct`, `earnings.surprise_beat_25pct`, `budget.block_100`), `nodes/{premarket,midday,postmarket,weekly_review}.py` (one LangGraph `StateGraph` per routine; nodes: `collect_facts → synthesize_brief → filter_alerts → publish_digest`), `report_pdf.py` (reportlab-based PDF builder for FR44), `prompts/{premarket,midday,postmarket,weekly_review}.md` (Jinja2 templates with bitemporal-fact placeholders), `models.py`, `repository.py`, `events.py`, `errors.py` (slice-local `IguanaError` subclasses).
- **APScheduler with SQLAlchemyJobStore** — `scheduler.py` instantiates `AsyncIOScheduler(jobstores={"default": SQLAlchemyJobStore(engine=engine)}, timezone=ZoneInfo("America/New_York"))`. Four cron jobs registered at app startup: `premarket` (cron: `day_of_week=mon-fri, hour=6, minute=30`), `midday` (`day_of_week=mon-fri, hour=12, minute=30`), `postmarket` (`day_of_week=mon-fri, hour=16, minute=30`), `weekly_review` (`day_of_week=sun, hour=18, minute=0`). All schedules ET-aware with DST handled by `zoneinfo` (Python 3.11+ stdlib).
- **LangGraph workflow per routine** — each `nodes/<routine>.py` defines a `StateGraph` with checkpointer (memory checkpointer MVP, persistent checkpoint v2 SaaS) and 4 nodes: `collect_facts` (queries `research_facts`, `trade_proposals`, `risk_evaluations`, `equity_snapshots` for the relevant window), `synthesize_brief` (calls `route_llm(TaskClass.ROUTINE_SUMMARY) → claude-3-5-haiku` by default, opus for weekly_review per O1 D2), `filter_alerts` (runs the routine's outputs through `alert_filter.classify_event` → emits tier-1 + accumulates tier-2 in digest), `publish_digest` (writes the digest payload to MessageBus event `orchestration.<routine>.digest_published` for P1 to pick up + persists `routine_runs.status='success'`).
- **`alert_filter.py` + `tier1_alerts.py` rules engine** — `alert_filter.classify_event(event) -> AlertTier` returns `TIER_1 | TIER_2 | TIER_3`. Tier-1 rules table is hardcoded data (not algorithm) in `tier1_alerts.py`: `risk.kill_switch.tripped → TIER_1`, `trading.ibkr.disconnected_90s → TIER_1`, `research.fda.approval_on_watchlist → TIER_1`, `research.insider.buy_pct >= 10 → TIER_1`, `observability.budget.block_100 → TIER_1`, etc. Tier-1 events trigger an immediate P1 channel emission (`MessageBus.publish("approval.alert.tier_1", payload)` consumed by `channels/telegram.py` + `channels/whatsapp_hermes.py`). Tier-2 events accumulate into the next routine digest. Tier-3 events go to `audit_log` only.
- **`alert_events` table** — every classified event persisted with `id`, `tenant_id`, `event_name`, `tier`, `routing_decision` (`emitted_to_channels: list[str] | "deferred_to_digest" | "audit_only"`), `payload JSONB`, `correlation_id`, `created_at`. Append-only via slice-3 `append_only_listener`. Indexed on `(tenant_id, tier, created_at)`. Subscribed by `api/sse/alerts.py` for the live W1 dashboard alert pane.
- **`routine_runs` table** — every cron execution logs `id`, `tenant_id`, `routine_name`, `scheduled_at` (cron-fired UTC ts), `started_at`, `ended_at`, `status` (`pending | running | success | timeout | error | skipped_budget | skipped_duplicate`), `duration_ms`, `cost_usd` (rolled up from `api_cost_events.routine_run_id` FK populated via `cost_meter` → routine_run contextvar), `digest_payload JSONB` (the published digest text + structured fields), `error_message`. Unique constraint `uq_routine_runs_routine_name_scheduled_at` enforces idempotency: duplicate triggers (e.g., scheduler restart fires the same cron twice) fail-fast with `IntegrityError` → handler logs `orchestration.routine.duplicate_trigger` + sets `status='skipped_duplicate'`.
- **LLM cost gating per routine** — `service.run_routine(name, scheduled_at)` calls `check_budget(tenant_id)` from O1 before invoking the LangGraph workflow. If `BLOCK_100`, the routine emits a "budget exceeded — skipping LLM synthesis, here is the deterministic fallback digest" message + status `skipped_budget`. If `WARN_80`, the routine proceeds but the routing layer auto-downgrades sonnet → haiku (already O1 D4 contract). Routine-level retry on transient LLM errors uses canonical backoff `[3, 6, 12, 24, 48]` (slice 2).
- **`api/sse/alerts.py`** — `GET /api/v1/stream/alerts` SSE endpoint streaming `alert_events` rows to authenticated clients, filtered by tenant + tier (default tier 1+2; admin can subscribe to tier 3). Slice-5 dynamic discovery picks the file up.
- **Migration `0007_orchestration_tables.py`** — creates `routine_runs` (per-tenant append-only-on-success path; running rows are mutable until `status` lands terminal — exception to append-only documented in design D6) + `alert_events` (per-tenant, append-only). `down_revision='0006_observability_tables'` per merge plan. Indexes per data-model: `ix_routine_runs_tenant_id_scheduled_at`, `uq_routine_runs_routine_name_scheduled_at`, `ix_alert_events_tenant_id_tier_created_at`, `ix_alert_events_event_name_created_at`.
- **`prompts/<routine>.md`** — Jinja2 templates with placeholders `{{ research_facts | format_facts }}`, `{{ open_positions }}`, `{{ today_earnings_calendar }}`, `{{ overnight_news_top_5 }}`, etc. Resolved by `nodes/<routine>.py::collect_facts` then rendered into the LangGraph state for `synthesize_brief`.
- **Tests** — `tests/integration/test_orchestration.py` with `freezegun` clock + APScheduler `executors={"default": MemoryJobExecutor()}` for deterministic cron firing. Cases: cron fires premarket at 06:30 ET on a weekday → routine_runs row inserted with `status='success'`, digest published; duplicate trigger → `status='skipped_duplicate'`; budget exceeded → `status='skipped_budget'` + fallback digest; tier-1 event during routine → emitted to P1 channels + `alert_events.tier=1` row; weekly_review generates PDF + emits with attachment metadata.

## Acceptance

- APScheduler `AsyncIOScheduler` boots at app startup with 4 cron jobs registered (`premarket`, `midday`, `postmarket`, `weekly_review`) using `SQLAlchemyJobStore` (NOT memory store) and `timezone=ZoneInfo("America/New_York")`.
- Each routine executes as a LangGraph `StateGraph` with 4 nodes (`collect_facts → synthesize_brief → filter_alerts → publish_digest`) and a memory checkpointer; LLM calls wrapped in `@cost_meter`; budget gated at routine entry via `check_budget(tenant_id)`.
- Tier-1 alerts emit via P1 Telegram/Hermes within 60s p99 (NFR-P3); tier-2 alerts accumulate into the next routine digest; tier-3 alerts persist to `audit_log` only.
- `routine_runs` rows persisted with start/end/status/duration_ms/cost_usd; SLA enforced at 5min p95 (NFR-P3); duplicate triggers fail-fast with `status='skipped_duplicate'`.
- Weekly PDF report generated via reportlab with equity curve + trades + costs + lessons + outlook (FR44); emitted via P1 channel as document attachment.
- DST transitions handled by `zoneinfo` (Python 3.11+ stdlib): `freezegun` test simulates 2026-03-08 (US spring-forward) + 2026-11-01 (fall-back) and asserts cron fires once per day, not zero or twice.
- All ISO 8601 UTC dates in `routine_runs.scheduled_at` / `started_at` / `ended_at`; structlog event names follow `orchestration.<entity>.<action>` per CLAUDE.md.
- `mypy --strict` clean on `contexts/orchestration/*` + `sse/alerts.py` + `dtos/orchestration.py` + `migrations/versions/0007_orchestration_tables.py`; pytest coverage ≥80% on the new context; pre-commit clean.
