# Proposal: dual-daemon-mode-toggle-and-reconcile

> **Split the single `trading_daemon` into parallel `trading_daemon_paper` + `trading_daemon_live` services with isolated IB Gateway sidecars; add a `tenant_trading_modes` table + `/api/v1/status` + `/api/v1/daemons/{mode}/toggle` + `/api/v1/daemons/{mode}/reconcile` endpoints; surface a persistent mode-aware chip in the web header; and add a reconcile-on-demand button in `/settings`.** This is the foundational slice that unblocks the paper→live promotion workflow and makes "which mode am I looking at?" answerable at a glance from any page.

## Why

Today the codebase has the *bookkeeping* for paper-vs-live trading (every `Trade`, `TradeProposal`, and `EquitySnapshot` row carries a `mode` column with CHECK constraint) but the *runtime* is single-mode: one `trading_daemon` boots with `--mode=paper|live` and stamps everything with that single value. Consequences today:

1. **Operator cannot run paper and live in parallel.** To validate a new strategy in paper while live is operating, the operator would have to stop live, switch the daemon to paper, and lose live coverage during the test. Most quants want concurrent operation: paper is the staging environment for live, and the two must coexist.
2. **Mode is invisible in the UI.** The roundtable on 2026-05-18 confirmed: `/proposals` shows `mode` as a muted grey badge in one column; `/portfolio`, `/risk`, `/trades`, `/research`, `/dashboard` show **nothing**. An operator cannot tell at a glance whether they're approving a paper trade or a live trade. This is exactly the failure mode that placed $200M of accidental real-money orders at Knight Capital in 2012 (different root cause, same class of "operator-confusion-on-mode" risk).
3. **No drain semantics.** There is no concept of "stop generating new signals but let in-flight orders complete." Today the only way to pause trading is to kill the daemon container, which leaves the operator unsure what happened to pending approvals and forces a hard restart to resume.
4. **No on-demand reconcile.** When the daemon restarts, IBKR's book may have moved (manual close, weekly restart, position exited at IBKR-side). There is no surfaced way to ask "tell me what IBKR currently holds and resync local state" without spelunking through `iguanatrader trading reconcile` CLI.
5. **Compliance / audit gap.** Per Diana (compliance) in the roundtable: a proposal generated by a paper-mode daemon should NOT be approvable-in-live (cross-mode contamination). Today the architecture does not prevent this because mode is daemon-level, not proposal-level-enforced.

This slice fixes all five gaps at once by lifting mode from a CLI flag to a first-class deployment concept with its own DB row, its own daemon process, and its own UI affordances.

## What

### 1. Compose split

`docker-compose.mvp.yml` and `docker-compose.ibgateway.yml` are restructured so the trading stack runs **two** daemons + **two** IB Gateway containers concurrently:

- `trading_daemon_paper` → `ib-gateway-paper` (port 4002, `TRADING_MODE=paper`)
- `trading_daemon_live` → `ib-gateway-live` (port 4001, `TRADING_MODE=live`)

Both daemons share the same DB volume (`iguanatrader_data`), the same `api` service, the same `openbb_sidecar` and ingest cron (research_facts ingestion is mode-agnostic — strategies + proposals + risk are mode-scoped). Each daemon has its own `IBKR_CLIENT_ID` (1 paper, 2 live) so IBKR-side concurrent sessions do not collide.

The live daemon **defaults to disabled** at first bring-up (the `tenant_trading_modes` flag is `live.enabled=false`). The container boots and idles — it reads the flag every loop tick and no-ops if `enabled=false`. Operator flips it via the UI when ready.

### 2. New table: `tenant_trading_modes`

Migration `0020_tenant_trading_modes.py`:

```python
op.create_table(
    "tenant_trading_modes",
    sa.Column("tenant_id", sa.UUID(as_uuid=True), nullable=False),
    sa.Column("mode", sa.String(8), nullable=False),  # 'paper' | 'live'
    sa.Column("enabled", sa.Boolean, nullable=False, server_default="false"),
    sa.Column("last_toggled_at", sa.DateTime(timezone=True), nullable=False, server_default=sa.func.now()),
    sa.Column("last_toggled_by_user_id", sa.UUID(as_uuid=True), nullable=True),
    sa.Column("reason", sa.Text, nullable=True),
    sa.PrimaryKeyConstraint("tenant_id", "mode"),
    sa.CheckConstraint("mode IN ('paper','live')", name="tenant_trading_modes_mode_chk"),
    sa.ForeignKeyConstraint(["tenant_id"], ["tenants.id"], ondelete="CASCADE"),
)
```

Both rows are seeded for every existing tenant by the migration: `(paper, enabled=true)`, `(live, enabled=false)`. Paper-by-default preserves current behaviour; live requires an explicit operator toggle.

### 3. New endpoints

**`GET /api/v1/status`** (session-auth, public to logged-in users):

```json
{
  "daemons": [
    {
      "mode": "paper",
      "enabled": true,
      "ib_connected": true,
      "last_heartbeat_at": "2026-05-18T15:42:11Z",
      "last_fill_at": "2026-05-18T15:39:58Z",
      "pending_proposals_count": 2
    },
    {
      "mode": "live",
      "enabled": false,
      "ib_connected": false,
      "last_heartbeat_at": null,
      "last_fill_at": null,
      "pending_proposals_count": 0
    }
  ],
  "fetched_at": "2026-05-18T15:42:14Z"
}
```

Backing reads:
- `enabled` ← `tenant_trading_modes` row
- `ib_connected`, `last_heartbeat_at` ← in-memory `DaemonHealthRegistry` updated by each daemon every 10s via a tiny `daemon_heartbeats` table (mode-keyed). Stale (>30s) → reported as `ib_connected=false`.
- `last_fill_at` ← `SELECT MAX(filled_at) FROM orders WHERE mode=:mode AND status='filled'`
- `pending_proposals_count` ← `SELECT COUNT(*) FROM trade_proposals WHERE mode=:mode AND state='pending_approval'`

**`POST /api/v1/daemons/{mode}/toggle`** (admin role):

Request:
```json
{ "enabled": true, "reason": "promoting validated donchian strategy to live", "password_reconfirm": "<user-password>" }
```

For `mode=live`, `password_reconfirm` is REQUIRED (re-entry of operator password verified against current session user). For `mode=paper`, it is optional. Server-side: the password is verified via the same hash-compare used at login; on mismatch return 403.

The endpoint:
1. Verifies caller has admin role.
2. Re-verifies password (live only).
3. Updates `tenant_trading_modes(tenant_id=current, mode=:mode)` with new `enabled`, `last_toggled_at=now()`, `last_toggled_by_user_id=current`, `reason`.
4. If toggling from `enabled=true` → `false` (drain): emits `BusEvent` `daemon.drain.requested(mode)` which the daemon listens for.
5. If toggling from `enabled=false` → `true`: emits `daemon.reconcile.requested(mode)` — the daemon runs reconcile BEFORE accepting new signals.
6. Writes structlog event `daemon.toggle` with mode + new state + user + reason (audit trail).
7. Returns the updated `tenant_trading_modes` row.

**`POST /api/v1/daemons/{mode}/reconcile`** (admin role):

Triggers an on-demand reconcile of the daemon's local state with IBKR's authoritative book. Reads from IBKR: positions, open orders, account cash. Writes to local DB: closes any local Trade rows that IBKR no longer holds, opens any Trade rows for positions IBKR has that we don't know about (with `provenance=ibkr_reconcile`), updates equity_snapshots with current IBKR-reported cash.

This endpoint can be called regardless of `enabled` state (you can reconcile a disabled daemon — useful before re-enabling).

### 4. Daemon drain + reconcile semantics

Both daemons gain a main-loop check at the top of each tick:

```python
async def tick(self):
    flag = await self.repo.load_trading_enabled(tenant_id, self.mode)
    if not flag:
        await self._handle_drain_if_pending()
        await self._heartbeat()  # keep DaemonHealthRegistry alive
        return  # no signal generation, no proposal emission, no order placement

    if self.pending_reconcile_request:
        await self._reconcile_with_ibkr()
        self.pending_reconcile_request = False

    await self._normal_tick()
```

`_handle_drain_if_pending()`:
1. Fetches all `trade_proposals` for this mode in state `pending_approval`.
2. Marks each as `state='rejected'`, `rejection_reason='daemon_drained'`, `rejected_at=now()`.
3. Does NOT cancel orders already submitted to IBKR — those continue their natural lifecycle (fill, cancel by IBKR, timeout). IBKR remains authoritative.
4. Idempotent: runs once per drain event, then sleeps until next toggle.

`_reconcile_with_ibkr()` is the on-demand reconcile logic (extracted from the existing CLI `iguanatrader trading reconcile`).

### 5. Frontend: persistent mode chips + reconcile button

**Layout header** (`apps/web/src/routes/(app)/+layout.svelte`):

Two chips, always visible, top-right of the header:

```
[PAPER · dinero ficticio]  [LIVE · dinero real]
   ↑ yellow tint              ↑ red/destructive tint
```

Visual states per chip:
- **dim/inactive**: low saturation background, `OFF` label → daemon `enabled=false` OR `ib_connected=false`
- **active**: full saturation background, `ON` label → both `enabled=true` AND `ib_connected=true`
- **pulse dot**: animates briefly (1s fade) when `last_fill_at` is within last 60s of `fetched_at` → "daemon just operated"
- **hover tooltip** (newbie disclosure): explains what paper/live mean, plus current daemon state in plain language

Polling: layout root has a `setInterval(fetch('/api/v1/status'), 5000)` — every 5s refresh. Polling pauses when document is hidden (tab inactive) to avoid waste.

Click on chip → modal:
- **Paper toggle**: simple "Activar paper / Desactivar paper?" + optional reason text → POST `/api/v1/daemons/paper/toggle`.
- **Live toggle**: stronger confirm — header `⚠️ ESTÁS A PUNTO DE ACTIVAR/DESACTIVAR LIVE TRADING (dinero real)` + reason text (required, min 20 chars) + password re-entry field → POST `/api/v1/daemons/live/toggle` with `password_reconfirm`. On 403 from server, show "contraseña incorrecta" and keep modal open.

**Settings page** (`apps/web/src/routes/(app)/settings/+page.svelte`) gains a §Daemons section with:
- Status table for both daemons (mode, enabled, ib_connected, last_heartbeat, last_fill, pending_proposals).
- Two buttons per daemon: `Reconcile ahora` (POST `/api/v1/daemons/{mode}/reconcile`) + `Toggle on/off` (same as chip click).
- Audit log preview: last 10 toggle events from `audit_events` filtered to `daemon.toggle`.

### 6. Proposal `mode` badge upgrade (UI polish)

`apps/web/src/lib/proposals/variants.ts` (or wherever the mode badge currently lives) gains explicit color mapping: `paper` → warning yellow, `live` → destructive red. The `/proposals` list column "mode" becomes a prominent badge (not muted grey). Tightly scoped — single file change, ~15 LOC.

### 7. Reconcile lifecycle wiring

Each daemon's startup sequence gains a reconcile-on-boot step (idempotent — if the previous reconcile completed cleanly there's nothing to do). This guarantees that no matter how the daemon died (graceful, crash, OOMkill), it always rebuilds a consistent view of IBKR before accepting new signals.

The same logic powers the on-demand button. Both paths share `_reconcile_with_ibkr()`.

## Out of scope (deliberately deferred to follow-up slices)

- **Per-mode strategy gating** (`Strategy.enabled_modes`) → separate slice `per-mode-strategy-gating` (roadmap O8). Enables "strategy X runs in paper but not live" without splitting strategy rows. Useful for paper→live promotion workflow but not blocking for dual-daemon launch.
- **Shadow mode** (Trade.state = 'shadow', live-book-against, no execution) → separate slice `trades-filters-and-shadow-mode` (roadmap U-next-4). Includes the generic filter component for `/trades`.
- **Strategy health observability** (last_run_at, last_error, signals_today, win_rate, etc.) → separate slice `strategy-health-observability` (roadmap O6).
- **Trade Order state timeline** (variants.ts 3-color fix + Order sub-state visualization on `/trades/[id]`) → separate slice `trade-state-and-order-timeline` (roadmap U-next-2).
- **Page-level borders / watermarks** distinguishing paper-view from live-view → cosmetic, deferred to U-next-3 if operator confusion persists post-launch.
- **Mode tag on every structlog event globally** → structlog binding contextvars work; the daemons + audit logger will tag it, but enforcing it in EVERY logger across the codebase is a follow-up tidy.
- **Multi-broker future** (one daemon per `broker × mode`) — current design generalises but only IBKR is wired today.

## Acceptance

- `docker compose -f docker-compose.mvp.yml -f docker-compose.ibgateway.yml up -d` brings up two daemons + two gateways. Both report healthy via compose healthcheck within 90s.
- Migration `0020` applies cleanly forward and reverses cleanly with `alembic downgrade -1`. Seeded rows for existing tenants: paper enabled, live disabled.
- `GET /api/v1/status` returns shape per §3. Smoke test verified by playwright/e2e.
- `POST /api/v1/daemons/live/toggle` with wrong `password_reconfirm` returns 403 with `code=password_mismatch`. Correct password + valid reason flips the flag and emits the bus event.
- Drain test: enable live, queue 3 pending_approval proposals, toggle live off → all 3 transition to `rejected` with `rejection_reason='daemon_drained'`. Existing IBKR-side orders are NOT cancelled.
- Reconcile test: pre-seed local trades that don't exist on IBKR-side; call `/api/v1/daemons/paper/reconcile` (uses fake IBKR adapter); local trades close with `provenance='ibkr_reconcile'`.
- Web header chip renders on every `(app)/*` page, polls every 5s, pauses on `document.hidden`, pulses on recent fills, opens correct modal on click.
- Live toggle modal blocks submission until password field is non-empty AND reason ≥ 20 chars.
- mypy --strict + ruff + black clean on touched files. Existing tests still pass. New tests (~25 added) green.

## Dependencies

- **No hard prerequisites.** Slice O1 (`sops-decrypt-at-boot`) is **soft prerequisite** — without it, the operator still manually exports `IBKR_USERNAME`/`IBKR_PASSWORD` for both paper and live. The dual-daemon split works either way; SOPS-at-boot reduces operator toil but does not gate this slice.
- **Requires a working live IBKR account.** Verified 2026-05-18 — operator has live account approved. Until live credentials are populated in `live.env.enc`, the live daemon boots with `enabled=false` and idles harmlessly.
- **PR #261 must be merged first** (digest pin + MCP token + roadmap baseline) — this slice builds on the compose structure it established.
