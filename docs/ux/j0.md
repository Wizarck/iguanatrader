# Journey 0 — Authentication & global error surfaces (DRAFT v1)

> Source: PRD §FR31 (auth flow), §FR38 (authorized-senders enforcement); Architecture Decisions §Authentication & Security (cookie-based JWT, HttpOnly + Secure + SameSite=Strict, 7-day sliding lifetime, Argon2id, slowapi 5/min); Architecture Decisions §Frontend Architecture (`+error.svelte` global error boundary, `(auth)/` route group, redirect-to with stable query params).

## 1. Goal

Cover **the gate** (everything between landing on the host and being inside the authenticated dashboard) and **global failure surfaces** (every state where SvelteKit's `+error.svelte` boundary takes over: 401 session expired, 404 not found, 500 internal error). J1+ assume the user is past J0; if any J1+ step needs an "unauthenticated" branch, it points back here.

J0 is a single-user MVP gate — no signup, no SSO, no MFA, no password reset UI. Reset is out-of-band (CLI). v3 SaaS revisits.

## 2. Trigger

- **Cold start**: Arturo opens `https://localhost:<port>/`; JWT cookie missing → redirected to `/login`.
- **Expired session**: any `(app)/*` route returns 401 from the API → SvelteKit hooks redirect to `/login?redirect_to=<current>`.
- **Stale link**: navigates to a route that doesn't exist (typo, deleted resource) → SvelteKit `+error.svelte` renders the 404 surface.
- **Backend hiccup**: API returns 5xx mid-render → `+error.svelte` renders the 500 surface with correlation_id.
- **Rate-limit reached**: 6th login attempt within 60s → form rerenders with rate-limit banner.

## 4. Components used

`(auth)/login/+page.svelte`, `+error.svelte`, `Input.svelte` (with error state), `Button.svelte` (primary + ghost + secondary + destructive variants), banner alert (inline composition of Card + icon), copy-to-clipboard widget. See [components.md](components.md) for full contracts.

## 5. Capabilities satisfied

- **Auth**: FR31 (cookie-based session, redirect-to, logout — logout itself stays in [j1.md](j1.md) Step 5 because it terminates an authenticated session, not initiates one).
- **Authorized senders**: FR38 (rate-limit 5/min via slowapi; only the email tied to JWT can reach the API surface — multi-channel guard at the API level, no UI surface per channel).
- **Resilience surfaces**: NFR-O2 (correlation_id always copyable on errors), NFR-R7 (retry affordance + filter preservation across re-auth).
- **Trust** (JTBD-6): every failure surface gives the user a way out (back / retry / home / sign-in-again). No dead ends.

## 6. Edge cases

- **JS disabled**: SvelteKit form actions still work (form POSTs server-side); `+error.svelte` renders SSR-only. Rate-limit countdown won't tick — banner stays until next form submit re-fetches.
- **Cookie present but expired (clock skew)**: server returns 401 → same flow as Step 2.
- **Cookie tampered**: signature verify fails → 401 → Step 2; correlation_id logged server-side, not exposed to user (avoid attacker feedback).
- **Multi-tab login race**: tab A signs in successfully, tab B was waiting on `/login`; tab B's next form submit succeeds without prompting again because cookie is now set. No special UI needed.
- **`+error.svelte` itself crashes** (e.g. SSR template error): SvelteKit falls back to its bundled barebones error page. We accept this — out of scope to retemplate the fallback.
- **Login attempted while authenticated**: form submit returns 200 with redirect to `/`; no special copy. (Keeps the form idempotent for muscle-memory submits.)

## 7. Decisions specific to J0

- **Login is its own route group `(auth)/`** — keeps the `(app)` authenticated layout clean (no Sidebar on login). Slice 4 owns the `(auth)/` directory; slice W1 owns `(app)/`.
- **`+error.svelte` is a single file** at the SvelteKit root, not duplicated per route group. The Sidebar conditionally renders only when the error happens inside `(app)/` (authenticated context); on `(auth)/` errors the page stands alone.
- **Correlation_id is always rendered as monospace** + copy button on 5xx, never on 4xx (4xx is user error and a corr-id leak adds noise without value).
- **No "Forgot password?" link** in the login footer — the copy is helper text inside a tinted box pointing at the CLI command. Reduces visual weight of an action that doesn't exist.
- **Brand mark in the login card** — placeholder "i" in the iguana teal accent block. Logo refresh is a separate decision, not blocking slice 4.
- **No password manager opt-out hints** — `autocomplete="email"` and `autocomplete="current-password"` set canonically; we don't fight the user's password manager.
- **Date format**: rate-limit countdown uses raw seconds ("Wait 47s before retrying"); 401 banner does not embed a timestamp (the user already knows it expired). When timestamps appear in error contexts (e.g. correlation ID generation time), they follow the project hard-rule ISO 8601 UTC + colocated relative.

## 8. Notes for implementation

- **Slice 4 (`auth-jwt-cookie`)** lands `(auth)/login/+page.svelte`, the JWT cookie middleware, the redirect-to behaviour with `redirect_to` allow-listing (must match same-origin paths), the form action, and the rate-limit banner.
- **Slice W1 (`dashboard-svelte-skeleton`)** lands `+error.svelte` (404 + 500 + 401 surfaces) and the global hooks for redirecting on 401. The 401 banner (Step 2) is rendered by the existing login page after slice 4 lands; hooks redirect logic is W1's responsibility.
- **rate-limit signal**: API returns `429` with `Retry-After: <seconds>` header; SvelteKit form action reads the header and seeds the countdown (no manual config in the page).
- **correlation_id propagation**: backend FastAPI middleware sets `X-Correlation-ID` on every response; SvelteKit `hooks.server.ts` reads it and exposes it to `+error.svelte` via `event.locals.correlation_id`. structlog event names: `auth.login.attempt`, `auth.login.failure`, `auth.login.rate_limit`, `error.boundary.render` (context = "auth" or "shared", entity = "login" or "boundary").
- **CSP headers**: `(auth)/` routes carry the same CSP as `(app)/` — no inline scripts. Login form is a SvelteKit form action, not JS-fetch.
- **A11y**: rate-limit banner has `role="alert"`; correlation_id row has `aria-label="correlation id, click to copy"`; error glyphs are decorative (`aria-hidden="true"`) since heading + body convey the meaning.
- **Theme**: J0 surfaces honour the same dark-first lock as J1+. No theme toggle on `(auth)/` — there's no Sidebar, and no settings affordance, so theme follows system preference only.
