# Journey 2 — Approval flow (DRAFT v1)

> Source: PRD §User Journeys / Journey 1 (the happy path: proposal → Telegram/WhatsApp → tap → fill confirm) + Journey 2 (edge case: cap warning → risk-rejection → override with reason ≥20 chars + double confirmation); FR12 (approve/reject within timeout window), FR13 (auto-discard on timeout), FR14 (broker submit), FR15 (lifecycle transitions), FR17 (force-exit), FR18 (force-create), FR24 (risk pre-rejection), FR25 (override with reason ≥20 chars + double confirm), FR26 (override audit), FR31-FR38 (channels + authorized senders + commands), FR48 (decision persisted with channel + latency), FR74 (proposal references `research_brief_id`). Personas: JTBD-1 (hands-off pipeline + veto power), JTBD-2 (programmatic risk policy enforcement).

## 1. Goal

Enable Arturo to receive a trade proposal **on whichever channel he is currently using** (Telegram or WhatsApp/Hermes are the messaging-first surfaces; web dashboard is the persistent surface), review it in ≤8 seconds, and approve / reject / override-with-reason — with the full decision and its rationale captured in the append-only audit trail (FR26 + FR48).

The journey covers both the happy path (Journey 1 in the PRD: 1-tap approve) and the edge-case override path (Journey 2 in the PRD: risk-rejected proposal escalated via written reason + double confirmation).

## 2. Trigger

- **Primary trigger**: bot detects a setup → strategy emits proposal → RiskEngine evaluates → if accepted, ApprovalService publishes to all authorized channels in parallel (FR32) AND to the dashboard SSE stream → Arturo's mobile vibrates + the dashboard `ApprovalCard` appears.
- **Edge-case trigger**: same flow, but RiskEngine pre-rejects (FR24) → proposal flagged `risk-rejected` and surfaces with `[Override]` affordance instead of `[Approve]`.
- **Force-create trigger** (FR18): Arturo issues `/force <symbol> <side> <qty>` from any approval channel → ApprovalService runs the same risk evaluation + emits a proposal that requires the same approval handshake.

## 6. Edge cases

- **User approves via Telegram while web `/approvals` is also open** → SSE pushes terminal state to web; `ApprovalCard` flashes "Decision recorded via Telegram" and collapses. No duplicate submit.
- **Network drop mid-approve on web** → optimistic UI reverts; toast with retry CTA; the underlying server state has not been mutated (POST never reached) so retry is safe. If POST reached but response lost, the server's idempotency guard (per FR48 audit + correlation_id) prevents double-execution; web shows "Decision already recorded" and converges via SSE.
- **Server SSE drops during `risk-rejected` review** → ConnectionIndicator banner; `OverrideForm` keeps local state; on reconnect, server snapshot reconciles. If the proposal expired during the drop, override modal closes with "Proposal expired during disconnect — re-evaluate or wait for next setup".
- **Two browser tabs racing the same approval** → first POST wins; second tab gets 409 and a toast "Already decided in another tab".
- **Override reason exactly 20 chars** → counter shows `--success`, submit enables. Whitespace counts (TBD: confirm rule with Sally — current draft says raw chars including whitespace).
- **Kill-switch fired while a proposal is pending** → all pending proposals auto-discarded with reason "kill-switch active"; web updates via SSE; no manual intervention required.
- **Authorized sender removed mid-session** (admin removes a phone from whitelist while that phone has an open approval window) → next command from that sender is silently dropped + logged (FR38). Web side is unaffected (different auth mechanism).

## 7. Decisions specific to J2

- **Optimistic UI for approve/reject** with SSE-backed reconciliation. Rationale: Arturo will have a noticeable latency (broker submit) — UI must not block.
- **OverrideForm is a modal, not a dedicated page**. Rationale: keeps the user in the approval context; preserves the `/approvals` filter/state; simpler return-to-list.
- **Double-confirmation pattern for override**: Step-1 reason → Step-2 read-only preview with a 1-second delay before Confirm becomes clickable. The 1-second delay is intentional friction (per FR25 spirit "this is NOT a routine operation").
- **Proposal preview includes `research_brief_id` chip** (FR74). The chip uses `CitationLink` (composed) so the visual is consistent with the research-domain citation language in J3.
- **Per-card keyboard shortcuts** (`A` approve, `R` reject, `O` override) only when the card has focus — no global hijack. Sally to confirm shortcuts.
- **Tier-2 cap-warning alert renders on `/risk`, not on `/approvals`** — separation of concerns: `/approvals` is per-proposal action, `/risk` is system-wide state.
- **Tier-2 / Tier-1 cap warning surface on `/risk`** = banner + timeline (locked 2026-05-05, validated against [variants/mock-j2-7-tier2-banner.html](variants/mock-j2-7-tier2-banner.html)). Banner sticky at top of the page while the threshold is breached (auto-dismisses when state drops below 80%; user "dismiss" hides it for the session but it reappears on a new threshold breach). Timeline section below the caps display shows chronological cap events for the last 24h (full history at a future `/risk/events` route, deferred). Tier-1 (cap fully consumed) reuses the banner shape with destructive tokens instead of warn-bg. The timeline row that mirrors the active banner anchors back to it via "↑ same event" cross-reference.

## 8. Notes for implementation

- **Slice P1 (`approval-channels-multichannel`)** lands the multichannel fanout including the DashboardChannel SSE writer + the 17 shared commands + authorized-sender enforcement + heartbeat resilience. The web view is consumer-only — P1 must guarantee the SSE event payload matches the `Proposal` interface in [components.md](components.md#31-approvalcardsvelte).
- **Slice T4 (`trading-routes-and-daemon`)** brings `/approvals/+page.svelte`, `/risk/+page.svelte` (overrides section), and `/trades/+page.svelte` (override-marker badge in history) online. ApprovalList + ApprovalCard + OverrideForm components are **owned by T4** (locked 2026-05-05 — Sally + Arturo). P1 stays backend-pure; W1 ships the dashboard skeleton + Lucide + tokens; T4 brings approval components online consuming the SSE contract P1 published. components.md treats them as canonical regardless of land-date.
- **Slice K1 (`risk-engine-protections`)** owns `risk_overrides` schema + `/risk` SSE events + the override server endpoint. The OverrideForm posts to `/api/v1/approvals/<id>/override`, but the server-side enforcement (reason ≥20 chars, risk_state_snapshot, confirmation_chain) lives in K1 territory.
- **SSE event types** consumed: `proposal.created`, `proposal.risk_rejected`, `proposal.timed_out`, `decision.approved`, `decision.rejected`, `decision.overridden`, `order.submitted`, `order.filled`, `order.failed` — typed via Pydantic models, generated to TS via openapi-typescript pipeline. Names must follow `<context>.<entity>.<action>` per `openspec-slice.md` Per-slice acceptance criteria.
- **Idempotency**: every approve/reject/override POST carries a client-generated UUID idempotency key in the header `Idempotency-Key`; server dedupes within a 60s window. Required to make optimistic UI safe against double-clicks + tab races.
- **Latency capture for FR48**: server records `decision_recorded_at - proposal_delivered_at_per_channel` per channel. The web channel's latency = SSE delivery round-trip + user-think-time. Used in the weekly review per J3.
- **A11y**: OverrideForm modal traps focus; `Esc` cancels; `aria-describedby` on the reason Input wires to the character counter. The 1-second delay on the Confirm button is announced via `aria-live="polite"` ("Confirm enabled in 1 second").
