Three options compared

Pure drawer (no URL state)

  • Closes → instantly back on filtered table; filter context preserved.
  • NOT deep-linkable: can't share a link to a specific trade; reload kills the drawer.
  • Browser back button doesn't close the drawer (UX surprise).
  • Implementation: simplest. Drawer is local state in +page.svelte.

Drawer + URL query (modal-route)

  • Drawer opens · URL becomes /trades?...&detail=tx_a3f2b1c.
  • Deep-linkable (paste URL → opens drawer over current filters); filter context preserved; back button closes drawer.
  • Pattern used by Linear / Notion / GitHub Issues — familiar to the user.
  • Implementation: SvelteKit $page.url.searchParams.get('detail') drives drawer state; goto with { keepFocus: true } avoids scroll jump.
  • Cost: ~10 extra LOC in +page.svelte vs pure drawer.

Dedicated /trades/[id] route

  • Full page on its own.
  • Deep-linkable + sharable; back button works natively; URL is canonical.
  • Filter state lost: clicking a row leaves /trades?range=last-90d&symbol=AAPL; pressing Back returns but causes a full page nav (felt as "I left and came back").
  • Workaround: persist filters in localStorage + restore on /trades mount → adds bug surface (stale localStorage, multi-tab races).
  • Implementation: extra route file /trades/[id]/+page.svelte + +page.server.ts.

My read after seeing it rendered

The drawer wins on filter-context preservation; the route wins on deep-linkability. The hybrid (drawer + URL query) gets both at ~10 LOC cost. Refresh the page on the URL with ?detail=tx_a3f2b1c and the drawer re-mounts; close the drawer and the URL drops detail while keeping range=last-90d&symbol=AAPL. The back button works as the user expects (one Back closes drawer, second Back exits filters).

→ Recommendation: Option A — drawer + URL query (modal-route hybrid)

Updates the J1 §3 contract: clicking a row pushes ?detail=<id> via SvelteKit goto({ keepFocus: true }). The drawer reads that param. Closing the drawer (Esc, X button, click backdrop) drops the param. Implementation lands in slice T4. The /trades/[id] route is NOT created — same content via drawer + deep-link query is cheaper and superior.
