Trade-offs side-by-side

Option A — inline rows

  • Cost: 5 LOC per flag (label + sub + Switch). No new component.
  • Information density: low. One-line label + one-line subtext. Good when the flag is self-explanatory.
  • Hindsight UX cost: subtext slightly cramped — "Strips outcome knowledge from research briefs to reduce hindsight bias" reads OK but doesn't surface the cost-multiplier or the side-effect (cache clear on toggle).
  • Future flags: copy/paste pattern. Each new flag = another row.
  • Migrate later: trivial. When 2nd flag justifies extra surface (warning, badge), promote to FeatureFlagToggle.svelte in one PR.
  • Risk: if FR81 needs the warning + badge + cost note (which it might, given Hindsight is non-trivial), Option A forces us to either drop those affordances or extend rows ad-hoc → inconsistency.

Option B — dedicated component

  • Cost: ~30 LOC component + N props per flag. Premature for a single flag.
  • Information density: high. Helper paragraph + experimental badge + optional warning row.
  • Hindsight UX cost: handles it well — badge says "experimental", helper text has cost multiplier, warning row says "cache clears on toggle". Future flags inherit the same surface.
  • Future flags: each new flag = new <FeatureFlagToggle /> with props. Consistent surface.
  • Risk: shape-locking the component before knowing flag #2's needs. If flag #2 needs a different affordance (e.g. a button "Reset to default" instead of a Switch), the component shape mismatches.
  • Mitigates: inconsistency across flags; the temptation to extend rows ad-hoc.

What changes my recommendation after seeing it rendered

Looking at the rendered Hindsight row in Option A: the subtext is too thin. "Strips outcome knowledge from research briefs to reduce hindsight bias" reads OK alone, but the user needs to know about the cost multiplier and the cache-clear side effect. To get those into Option A's surface, we'd extend the row pattern — which inverts the YAGNI argument because we'd be ad-hoc'ing the component anyway.

After rendering: I revise toward Option B for FR81 specifically. The flag carries enough nuance (cost, side-effect, experimental status) that pretending it fits in a one-line row is wishful thinking. Promoting to FeatureFlagToggle.svelte on day 1 is cheap (30 LOC) and locks the right surface for future flags.

→ Revised recommendation: Option B

Ship FeatureFlagToggle.svelte with FR81 in slice R6 (hindsight-integration). The Hindsight flag's three affordances (badge, helper paragraph with cost note, optional warning row) define the canonical shape; future flags either fit the shape or become a new component with its own decision.
