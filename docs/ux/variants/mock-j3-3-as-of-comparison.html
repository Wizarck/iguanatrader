Three options compared

A — Side-by-side

  • Two columns: as-of (left, warn-bg label) vs current (right, accent label).
  • Inline highlights inside each column show what's added (success-tinted) / removed (destructive-tinted) / changed (warn-tinted).
  • Best for spatial recall: "what did I see when I made the trade?" → eyes go left; "what do I see now?" → eyes go right.
  • Works well across all content types: prose, fact tables, numerical deltas all fit the 2-col grid.
  • Tradeoff: doubles horizontal space. On narrow widths (≤1200px) becomes cramped — mock-tested at 480px-per-col, fine on desktop.
  • Familiar from doc-comparison tools (Google Docs, Word, Notion versioning).

B — Overlay (unified)

  • Single column; old + new merged with inline highlight markers.
  • Compact horizontally; great on narrow widths.
  • Loses the "where did this come from" mental anchor — the user sees "$850 → $920" but has to read the legend to know which is which.
  • Prose with many removals becomes a strikethrough soup.
  • Familiar from Google Docs Suggesting mode.

C — Diff (git-style ±)

  • Line-prefix markup: − / + / context.
  • Engineer-familiar; precise for prose; perfect for code-like content.
  • Heavy mono font + chunky vertical lists feel out of place for a research brief — Arturo is reading analysis, not code.
  • Best for AuditTrailViewer (a future surface where calculations are versioned), not for the brief-vs-brief use case.
  • Would need extra UI to navigate the hunks (jumps).

My read

A wins on the actual user task: Arturo is post-mortem-ing a trade and his mental model is "before / after". A 2-col layout matches that frame. The inline highlight legend (added / removed / changed) inside each column gives him the granularity without forcing him to translate −/+ to "as-of vs now". B is more compact but loses the spatial anchor. C reads like a git diff — fine for engineers reviewing calculations (which is where I'd reuse this pattern, in a future AuditTrailViewer surface) but heavyweight for a research brief.

→ Recommendation: Option A — side-by-side (default), with toggle to overlay

Default view = side-by-side; toggle bar at top of the page (`Side-by-side` / `Overlay`) lets the user switch when columns get cramped (mobile / tablet auto-collapses to overlay). Drop the diff option for now — it's not the right shape for prose. A future AuditTrailViewer surface (slice R5) for calculation-trail comparison may justify a dedicated diff view, but that's a separate decision (own mock when slice R5 lands). Inline highlight tokens: added=success-tinted, removed=destructive-tinted strikethrough, changed=warn-tinted. Implementation lands in slice R5 (research-brief-synthesis) on the /research/[symbol]?as_of=<ts> route.
