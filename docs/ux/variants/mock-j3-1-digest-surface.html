Four options compared

A — dedicated route

  • Each digest = its own URL.
  • Deep-linkable, shareable.
  • No archive view — just the per-digest page; user has to navigate elsewhere to find the list.
  • Going to the digest from /portfolio = full nav; back button leaves the dashboard surface.

B — modal interrupt

  • Digest fires while user is mid-task → modal pops over.
  • Premarket scan is NOT urgent — it's informational; modal interruption is annoying.
  • Closing dismisses it; no easy way to revisit.
  • Not deep-linkable; modal state evaporates.
  • Best fit for blocking decisions (kill-switch confirmations); wrong fit for routine output.

C — banner on dashboard home

  • Sticky at top of /; collapsible.
  • Lost when user leaves the home page (banner only renders on /).
  • Multiple routines stack the banners → clutter on busy days (premarket + midday + postmarket = 3 banners).
  • No archive of past digests beyond what's still active.

D — modal-route over /alerts list

  • /alerts = chronological archive route.
  • Each digest reached via ?digest=<id> drawer (modal-route pattern from §0.4b).
  • Sidebar badge "Alerts (3)" = unread count, persistent.
  • Deep-link works (/alerts?digest=<id>); sharing works; back button works.
  • Archive + reading + freshness in one surface; no per-page banner clutter.
  • Reuses Drawer + Modal-route pattern just locked → catalogue consistency.
  • Cost: same as A (one route file) but cleaner because the route is the LIST, not the per-digest page.

My read

A and D both create a dedicated surface; the difference is whether the route is the digest itself (A) or the list with a drawer (D). D wins because (a) the user usually wants the archive — "what was the postmarket reconciliation last Wednesday?" — and that requires a list view anyway; (b) modal-route preserves cross-page filters consistent with J1-3; (c) one surface (/alerts) instead of two (/alerts list + /alerts/[id] per-digest). B is wrong shape (digests aren't urgent). C alone fails on cross-page visibility + history.

→ Recommendation: Option D — modal-route over /alerts list

/alerts = chronological archive (filterable by routine type, date). Each digest opens in a drawer via ?digest=<id> per the modal-route pattern (components.md §0.4b). Sidebar badge shows unread count; clearing happens when the drawer for that digest is opened. NO modal interruption (digests are informational, not urgent). NO banner on / dashboard (avoids per-routine-type banner clutter). Toast on dashboard if user is online when a digest fires (small, dismissable, with a "View" CTA that opens the drawer). Implementation lands in slice O2 (orchestration-scheduler-routines) which owns the routine pipeline; /alerts route + drawer in slice W1 / O2 split (TBD with slice authors).

Note: Toast for "digest just arrived" is a separate surface (Toast.svelte primitive, already canonical in components.md §5). It's complementary to the drawer — Toast = "hey, premarket just landed, click View to read it"; drawer = "now read it". Toast auto-dismisses after 8s; clicking View pushes ?digest=<id>. If the user is on /alerts when the digest fires, no Toast — the list updates inline and the new row gets the unread border-left.
